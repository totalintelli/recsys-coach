# 포트폴리오 — 커머스 구매 예측 추천 시스템

> 면접·발표용. 기술 선택의 **근거와 시행착오** 중심. 수치는 EXPERIMENT_LOG.md 참조.

## 한 줄 요약

638,257명에게 다음 구매 상품 10개를 추천하는 대회에서, "정답의 본질은 재구매"라는 데이터 관찰을 근거로
협업필터링(ALS) 대신 co-visitation 기반 모델을 선택했고, 제출 파이프라인을 약 3시간에서 4분으로 단축했다.

---

## 1. 평가지표를 코드보다 먼저 읽다 (가장 큰 교훈)

**상황:** 로컬 CV로 covisit이 NDCG@10 0.31이 나와 좋은 모델이라 판단, 제출했더니 **LB가 0.1544**로 절반이었다.

**진단:** 점수 부풀림을 의심하고 대회 평가방법 PDF를 직접 읽었다. PDF 예시에서 IDCG가 "정답 수와 무관하게
k개 위치를 모두 1로 채우는(full_k)" 방식임을 확인 — 내 CV는 표준 NDCG(정답 수만큼만 채움)를 쓰고 있었다.
같은 추천이라도 이 차이로 covisit이 0.31(standard) vs 0.073(full_k)로 **4배** 달랐다.

**행동:** CV 기본 IDCG 모드를 full_k로 교정. 이후 모든 모델 비교를 LB와 같은 기준으로 수행.

**결과/교훈:** "내 CV가 대회 채점식과 같은가"를 코드 짜기 전에 검증해야 한다. 지표 정의 하나가
모든 의사결정의 나침반을 바꾼다. (남은 CV-LB 2배 차이는 검증 주와 test 주의 구매자 분포 차이로 추정 —
train만으론 검증 불가라 단정하지 않고 "상대 순위는 유효한 나침반"으로 활용.)

## 2. 데이터가 모델 선택을 결정하다 (ALS를 데이터로 기각)

**상황:** 베이스라인으로 제공된 ALS(implicit 협업필터링) 제출본을 쓸지 검토.

**진단:** 정답 유저의 구매 아이템 대부분이 **이미 view/cart 이력이 있는 아이템(재구매)**임을 확인.
ALS는 잠재요인으로 "안 본 아이템"을 발굴하는 모델이라 방향이 어긋난다고 가설. 검증을 위해
ALS를 ① 이벤트 가중 ② alpha 조정 ③ factors 확대 ④ covisit과 RRF 블렌딩까지 네 갈래로 측정.

**행동/결과:** 모두 covisit(0.0731) 미달. ALS 최고가 0.0462(시간감쇠), 블렌딩은 ALS 비중을 줄일수록
covisit 단독으로 수렴(섞으면 무조건 손해). Recall도 ALS 0.29 vs covisit 0.42로 정답을 덜 건졌다.

**교훈:** "유명한 모델"이 아니라 "데이터의 정답 구조에 맞는 신호"를 쓰는 모델이 이긴다. 가설은
직관으로 세우되 데이터로 기각한다. 가능성을 천장까지 확인하고 미련 없이 접는 것도 의사결정이다.

## 3. 병목은 측정으로 찾는다 (GPU가 아니라 알고리즘)

**상황:** 제출 파일 생성이 covisit에서 약 3시간 걸려 실험 반복이 막혔다. "GPU를 쓰면?"이 첫 발상.

**진단:** 코드를 보니 병목이 행렬곱(GPU가 빠른 연산)이 아니라 pandas groupby + 파이썬 dict 누적
루프(CPU·단일코어 바운드)였다. GPU는 무효. 단계별 타이밍으로 진짜 병목을 좁히니, 인기 아이템의
co-visitation 이웃이 최대 2.4만 개라 유저마다 그걸 순회하는 안쪽 루프가 비용의 대부분이었다.

**행동:** ① dict 누적을 정수 코드 매핑(factorize) + numpy 누적으로 벡터화 ② 이웃을 가중 상위
100개로 컷(top_nbr=100). 컷이 NDCG를 떨어뜨리지 않음(오히려 노이즈 이웃 제거로 미세 상승)을 홀드아웃으로 검증.

**결과:** 약 3시간 → **264초(4분 24초)**. GPU·멀티프로세싱 없이 알고리즘 컷과 벡터화만으로 달성.

**교훈:** 최적화 전에 측정. 그리고 covisit은 이웃 크기가 데이터 양에 비선형이라 표본 추정이 반복적으로
빗나갔다 — 표본으로 추정하지 않고 전체 실행으로 확인하는 규율이 필요했다.

## 4. `[object Object]` 버그 — 증상이 아니라 구조를 고치다 (디버깅)

**상황:** Q&A 답변에 코드 블록이 나오면 화면이 `[object Object]`로 가득 찬 회색 박스로 깨졌다.
앞선 두 커밋에서 "중첩 `<pre>` 제거", "빈 코드펜스 제거" 같은 **증상 패치**를 넣었지만 박스는 계속 나왔다.

**진단:** 가설 3개를 세워 하나씩 기각했다.
① LLM 출력 자체에 노이즈가 섞였다 → `_clean_answer`로 이미 정제 중, 입력은 깨끗했다.
② 생성된 HTML이 잘못됐다 → 함수 출력을 직접 찍어보니 `<pre><code>`까지 **정상 HTML**이었다(기각).
③ 렌더러가 깨뜨린다 → 남는 건 이것뿐. `[object Object]`는 **JavaScript가 객체를 문자열로 강제 변환할 때** 나오는 시그니처다.
범인은 `st.markdown(html, unsafe_allow_html=True)` — Streamlit은 주입된 HTML을 자체 React 파이프라인으로 **다시 파싱**하는데,
인라인 스타일이 박힌 코드 블록을 React 노드로 변환하지 못하면 `[object Object]`로 떨어뜨린다. HTML은 멀쩡했고, 렌더러가 병목이었다.

**행동:** HTML을 주입하던 165줄짜리 `_md_to_html`(markdown-it-py + pygments로 마크다운→HTML 재구현)을 **통째로 삭제**.
Streamlit의 `st.markdown(text)`는 펜스 코드·표·목록을 **기본으로 렌더링**한다 — 우리가 재구현한 그 기능 전부를. 호출부 2곳을 한 줄로 교체.
LLM이 뱉는 `[object Object]` 노이즈를 제거하는 `_clean_answer`는 답변 경로에 독립적으로 걸려 있어 그대로 유지.

**결과:** 버그 해결과 동시에 **net −113줄**(추가 170 / 삭제 283), 의존성 2개(markdown-it-py, pygments) 제거.
같은 노이즈가 재발하지 않도록 LLM 출력 패턴 **회귀 테스트 5건**을 추가(빈 펜스, 미닫힘 펜스, 리스트 내부 펜스 등). 전체 38개 테스트 통과.

**교훈:** 버그 리포트는 증상의 이름일 뿐이다. 증상이 난 경로만 패치하면 형제 경로는 그대로 깨져 있다.
근본 원인(렌더러)을 찾으면 **고치는 코드가 오히려 더 작다** — 한 곳을 지우는 게 여러 곳을 땜질하는 것보다 작은 diff였다.
그리고 프레임워크가 이미 하는 일을 재구현하지 않는 것(YAGNI)이 버그의 표면적 자체를 줄인다.

## 예상 면접 질문 & 답변

**Q. 추천 모델로 왜 딥러닝/협업필터링이 아니라 co-visitation을 썼나요?**
정답의 본질이 재구매(이미 본 아이템)였기 때문입니다. ALS를 가중·블렌딩까지 검증했지만 covisit의 63%에
그쳤습니다. 데이터의 정답 구조에 맞는 신호를 쓰는 게 모델 정교함보다 우선이라 판단했습니다.

**Q. CV 점수와 리더보드가 2배 차이 났는데 어떻게 대응했나요?**
먼저 평가식을 의심해 PDF를 읽고 IDCG 정의(full_k)를 바로잡아 4배 차이를 제거했습니다. 남은 2배는
검증 기간과 test 기간의 구매자 분포 차이로 추정되며 train만으론 검증 불가라 단정하지 않았습니다.
대신 "절대값은 어긋나도 모델 간 상대 순위는 보존된다"를 확인해 CV를 비교 도구로 계속 신뢰했습니다.

**Q. 3시간짜리 작업을 4분으로 줄인 핵심은?**
병목을 측정으로 찾은 것입니다. GPU가 첫 발상이었지만 병목이 행렬곱이 아니라 파이썬 dict 루프라
무효였고, 인기 아이템의 거대한 이웃 순회가 진짜 원인이었습니다. 이웃을 상위 100개로 자르고(품질 검증 후)
numpy로 벡터화해 해결했습니다.

**Q. 정확도를 바꾸는 최적화(이웃 컷)를 어떻게 안전하게 적용했나요?**
컷 적용 전후 NDCG를 홀드아웃으로 비교해 떨어지지 않음을 확인한 뒤에만 적용했습니다. 속도가 정확도를
깎으면 채택하지 않는다는 원칙을 지켰습니다.

**Q. UI 버그(`[object Object]`)를 어떻게 디버깅했나요?**
앞선 패치들이 증상만 건드렸기에, 가설 3개(LLM 출력 / HTML 생성 / 렌더러)를 세워 증거로 기각했습니다.
함수 출력 HTML이 정상임을 직접 확인해 범인을 렌더러로 좁혔고, `[object Object]`가 JS의 객체→문자열 강제변환
시그니처라는 점에서 Streamlit이 주입 HTML을 재파싱하다 깨뜨린다고 결론지었습니다. 해법은 HTML을 재구현한
165줄을 지우고 Streamlit 기본 마크다운 렌더링을 쓰는 것이었고, 버그 수정과 동시에 코드·의존성이 줄었습니다.

**Q. 코드를 줄이는 결정은 어떤 기준으로 하나요?**
"프레임워크/표준 라이브러리가 이미 하는 일인가"를 먼저 봅니다. `_md_to_html`은 Streamlit `st.markdown`이
기본 제공하는 마크다운 렌더링을 165줄로 재구현한 것이라, 중복일 뿐 아니라 버그의 원인이기도 했습니다.
재구현한 코드는 유지보수 비용과 버그 표면적을 둘 다 늘립니다 — 그래서 지우는 쪽을 택했습니다.

## 부록 — RAG 파이프라인 아키텍처 (Q&A 기능)

대회 문서 Q&A는 단순 벡터검색이 아니라 **하이브리드 검색 + 재랭킹** 구조로 설계했다.

```
문서 → 청킹(600자/80자 오버랩) → 출처 헤더 prefix → Upstage 임베딩 → FAISS
질문 → BM25 + FAISS 앙상블 검색(weights 0.4/0.6, k=6)
     → Cross-Encoder 재랭킹(ms-marco-MiniLM-L-6-v2) → 상위 3청크
     → LLM 생성(Solar Pro / 로컬 Qwen·Llama) → 답변 + 출처
```

**설계 근거:**
- **BM25+FAISS 앙상블** — 의미검색(FAISS)만으론 고유명사·약어·정확한 토큰 매칭을 놓쳐, 키워드검색(BM25)을 0.4 비중으로 섞었다.
- **Cross-Encoder 재랭킹** — 앙상블이 넓게 6개를 가져오면 Cross-Encoder가 질문-청크 쌍을 직접 채점해 상위 3개로 정밀화. 리트리버 캐싱·모듈 캐시로 질문마다 재로딩하는 비용 제거.
- **백엔드 추상화** — `LLM_BACKEND` 환경변수로 Upstage/Qwen/Llama/vLLM 분기, `OFFLINE_MODE`로 API 없이 검색 컨텍스트만 반환(개발·테스트). `langchain.chains.RetrievalQA` 대신 LCEL로 직접 조립해 각 단계를 제어 가능하게 구성.

Python, pandas, numpy(벡터화), scipy.sparse, implicit(ALS), MLflow(실험 추적), tqdm.
LangChain(LCEL), FAISS, BM25, Cross-Encoder(sentence-transformers), Streamlit, Upstage Solar API.
