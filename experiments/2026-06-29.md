## 변경 내역

### 1. OFFLINE_MODE에서 Upstage 임베딩 API 호출 차단

**문제**: `OFFLINE_MODE=true`로 설정해도 `build_vectorstore()`가 Upstage Embeddings API를
호출해 오프라인 개발 의도를 위반하고 API 크레딧을 소모했다.

**수정**: `_get_embeddings()` 함수에 분기 추가.

```python
# src/qa_chain.py
def _get_embeddings():
    if OFFLINE_MODE:
        from langchain_huggingface import HuggingFaceEmbeddings
        return HuggingFaceEmbeddings(
            model_name="sentence-transformers/paraphrase-multilingual-MiniLM-L12-v2",
            model_kwargs={"device": "cpu"},
        )
    from langchain_upstage import UpstageEmbeddings
    return UpstageEmbeddings(api_key=UPSTAGE_API_KEY, model="solar-embedding-1-large")
```

**선택 모델**: `paraphrase-multilingual-MiniLM-L12-v2`
- 한국어 포함 50개 언어 지원
- `sentence-transformers`가 reranker 의존성으로 이미 설치되어 있어 추가 패키지 불필요
- 모델 크기 ~420MB, CPU 추론 가능

### 2. OFFLINE_MODE 사용자 안내 배너 추가

**문제**: 오프라인 모드에서는 LLM 생성 없이 검색 결과를 raw 텍스트로 반환하는데,
사용자에게 이 사실이 표시되지 않아 답변 품질이 낮은 이유를 알 수 없었다.

**수정**: Q&A 탭 상단에 `st.info` 배너 추가. 임베딩 모델명과 동작 방식을 명시.

```python
# app.py
if _OFFLINE:
    st.info(
        "**오프라인 모드** — Upstage API를 호출하지 않습니다. "
        "임베딩은 로컬 모델(`paraphrase-multilingual-MiniLM-L12-v2`)을 사용하고, "
        "답변은 LLM 생성 없이 검색 결과를 직접 반환합니다.",
        icon="ℹ️",
    )
```

벡터 인덱스 생성 단계에도 사용 중인 임베딩 종류를 스피너 메시지로 표시:

```
검색용 벡터 인덱스를 생성하는 중입니다. 로컬 임베딩 모델 사용 중.
```

### 3. 모델 웜업 진행 상황 실시간 표시

**문제**: `warm_up_llm()` 내부에서 cross-encoder reranker 모델이 로딩될 때
(첫 실행 시 ~5~10초 소요) 스피너에 "LLM 모델 준비 중..."만 표시되어
어느 단계에서 시간이 걸리는지 알 수 없었다.

**수정**: `warm_up_llm()`에 `progress` 콜백 파라미터 추가.
앱에서 `status.update()`를 콜백으로 전달해 단계별 상태를 실시간 갱신.

```python
# src/qa_chain.py
def warm_up_llm(model=None, progress=None):
    def _step(label):
        if progress is not None:
            progress(label)

    _step("리랭커 모델 로딩 중 (cross-encoder)...")
    _rerank_with_cross_encoder(...)

    if not OFFLINE_MODE and backend in _LOCAL_BACKENDS:
        _step(f"LLM 모델 로딩 중 ({backend})...")
        _get_llama_llm()
```

```python
# app.py
warm_up_llm(
    st.session_state.get("selected_model"),
    progress=lambda label: status.update(label=label, state="running", expanded=True),
)
```

### 4. Q&A 문서 업로드에 Markdown(.md) 지원 추가

**문제**: 파일 업로더가 `type=["pdf", "txt"]`만 허용해 대회 문서로 흔한
`.md` 파일을 올릴 수 없었다.

**수정**: 허용 확장자에 `md` 추가. MD는 plain text라 기존 TXT 처리 분기
(`raw.decode("utf-8")`)가 그대로 처리하므로 로직 변경 불필요.


### 5. 구분점 목록 렌더링 오류 수정

**문제**: LLM이 제목·단락 직후 빈 줄 없이 목록을 이어서 생성하면
(`## 접근 전략\n- 방법 A`), Streamlit CommonMark 렌더러가 이를 bullet list로
파싱하지 못하고 `- 방법 A`를 일반 텍스트로 출력했다.

**수정**: `_clean_answer()`에 정규식 두 줄 추가. 비목록 줄 다음에 목록 항목
(`-` / `*` / `+` / `1.`)이 바로 이어지면 빈 줄을 삽입한다 (unordered·ordered 모두).

```python
# src/qa_chain.py
# 제목·단락 직후 목록 항목 앞 빈 줄 보장
text = re.sub(r"(?m)(?<=[^\n])\n([ \t]*[-*+] )", r"\n\n\1", text)
text = re.sub(r"(?m)(?<=[^\n])\n([ \t]*\d+\. )", r"\n\n\1", text)
```

중첩 목록(`  - 하위 항목`)의 들여쓰기는 보존되어 계층 구조가 유지된다.

### 6. 물결(~) 취소선 오변환 수정

**문제**: Streamlit CommonMark 렌더러는 한 줄에 `~`가 둘 이상이면 그 사이를
`~~취소선~~`으로 해석한다. 한국어 범위·근사 표기(`5~10초`, `~420MB`, `32~64`)가
여기에 걸려 의도치 않은 취소선으로 렌더링됐다.

**수정**: `_clean_answer()`에 `_escape_literal_tildes()` 추가. 코드 영역
(코드 블록·인라인 코드)은 마크다운이 적용되지 않으므로 건드리지 않고,
본문의 `~`만 `\~`로 이스케이프해 문자 그대로 렌더링되게 한다.

```python
def _escape_literal_tildes(text: str) -> str:
    # 코드 펜스·인라인 코드를 캡처로 분리 → 짝수 인덱스(코드 밖 본문)에서만 ~ 이스케이프
    parts = re.split(r"(```.*?```|`[^`\n]+`)", text, flags=re.DOTALL)
    for i in range(0, len(parts), 2):
        parts[i] = parts[i].replace("~", "\\~")
    return "".join(parts)
```

검증: 본문 `5~10초` → `5\~10초`(정상), 인라인 코드 `arr[~0]`·`~/data` 보존,
코드 블록 `x = ~mask` 보존.
